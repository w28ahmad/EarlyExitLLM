# MLP Early-Exit Predictors (Wahab)

Training and evaluation pipeline for MLP predictors used in the [SpecEE](../SpecEE/SpecEE_cloud/) early-exit system on LLaMA-2-7B. These predictors decide, at each decoder layer, whether the model can safely skip remaining layers and exit early during speculative decoding.

## Background: The Problem

SpecEE deploys MLP predictors at a dynamic subset of decoder layers (default: `[15, 16, 17, 19, 21, 24, 26, 28]`). Each predictor receives a 12-feature vector:

| Features | Description |
|----------|-------------|
| `logit_0..3` | Top-4 draft logits from the draft model's LM head projection |
| `prob_0..3` | Softmax of the top-4 draft logits |
| `gap_0..3` | `prob[current_layer] - prob[last_evaluated_layer]` |

The **gap features** depend on which layers are evaluated. At runtime, SpecEE's dynamic scheduler skips layers, so `gap` may span 2-3 layers (e.g., `prob[19] - prob[17]`). However, our training data (`01_naive_all_layers/`) was generated by evaluating **every** layer contiguously, so `gap` is always between consecutive layers (e.g., `prob[19] - prob[18]`).

This creates a **distribution mismatch** between training and inference for the gap features.

## Our Approaches

We train two sets of predictors to study this mismatch:

### Approach 1: Naive Contiguous Baseline

Use all 12 features exactly as generated, with contiguous gap values. This is the simplest baseline and ignores the train/inference mismatch. The hypothesis is that the gap signal between consecutive layers is close enough to non-consecutive gaps that the MLP generalizes.

### Approach 3: Augmented Dynamic Gaps

Augment the training data by recomputing gap features at multiple skip distances (1, 2, and 3 layers back), simulating the range of gaps the predictor will see at runtime. This produces **3x the training data** per layer. The skip distances were chosen to cover the actual gaps in SpecEE's default schedule:

| Schedule Layer | Previous in Schedule | Skip Distance |
|:-:|:-:|:-:|
| 15 | (first) | gap = prob |
| 16 | 15 | 1 |
| 17 | 16 | 1 |
| 19 | 17 | 2 |
| 21 | 19 | 2 |
| 24 | 21 | 3 |
| 26 | 24 | 2 |
| 28 | 26 | 2 |

The augmentation is possible because rows are aligned across layer CSVs within each dataset (row *i* in `layer_5.csv` and `layer_3.csv` correspond to the same input token), so we can look up the prob values at any prior layer and recompute the gap.

## Architecture

Matches the original SpecEE predictors exactly:

```
MLP(input_size=12, hidden_size=512, output_size=1)
  fc1: Linear(12, 512) + ReLU
  fc2: Linear(512, 1)  + Sigmoid
```

Output is a scalar in `[0, 1]` — the probability that the model can safely exit at this layer. If `output > threshold` (default 0.5), SpecEE exits early.

## Training Details

| Parameter | Value |
|-----------|-------|
| Datasets | alpaca + gsm8k + commonsenseqa + sst2 (combined) |
| Samples per layer | ~127K (approach 1), ~380K (approach 3, 3x augmented) |
| Train/Val split | 80/20, stratified by label |
| Loss | BCELoss with class weighting (upweight positives) |
| Optimizer | Adam, lr=1e-3 |
| LR scheduler | ReduceLROnPlateau(patience=3, factor=0.5) |
| Early stopping | patience=5 on validation loss |
| Max epochs | 50 |
| Batch size | 1024 |
| Device | CPU |

**Class imbalance handling:** Early layers have very few positive labels (e.g., 2.3% at layer 0, 0% for small datasets). We compute `pos_weight = n_negative / n_positive` and scale positive sample losses accordingly. Layers with zero positives save an untrained model (they should never trigger early exit anyway).

## Comparison: Approach 1 vs Approach 3

Schedule layers `[15,16,17,19,21,24,26,28]` at threshold=0.5:

| Metric | Approach 1 (Naive) | Approach 3 (Augmented) | Winner |
|--------|:------------------:|:----------------------:|:------:|
| Avg F1 | 0.7667 | **0.7748** | A3 (+0.8%) |
| Avg Accuracy | 0.8407 | **0.8443** | A3 (+0.4%) |
| Avg Precision | 0.7151 | **0.7227** | A3 (+1.1%) |
| Avg Recall | **0.8580** | 0.8626 | A3 (+0.5%) |
| Avg Exit Rate | 42.5% | 42.3% | ~Same |
| Training Time | **10.5 min** | 37.5 min | A1 (3.6x faster) |

**Takeaway:** Approach 3 shows a modest improvement across all metrics (+0.4-1.1%), suggesting the augmented gaps do help the MLPs generalize to non-contiguous scheduling. However, the improvement is small — the naive contiguous gaps are already a reasonable approximation. The real test is end-to-end inference on a GPU, where the dynamic scheduling mismatch is actually in play.

## Usage

### Training

```bash
cd EarlyExit
source .venv/bin/activate

# Approach 1 (naive baseline)
python mlp_wahab/train.py --approach 1 \
    --data-root ./01_naive_all_layers \
    --output-dir ./mlp_wahab/trained_models/approach1

# Approach 3 (augmented gaps)
python mlp_wahab/train.py --approach 3 \
    --data-root ./01_naive_all_layers \
    --output-dir ./mlp_wahab/trained_models/approach3

# Train a single layer (for quick validation)
python mlp_wahab/train.py --approach 1 --layer 15 \
    --data-root ./01_naive_all_layers \
    --output-dir ./mlp_wahab/trained_models/approach1
```

Monitor progress: `tail -f mlp_wahab/trained_models/approach1/training_log.txt`

### Evaluation

```bash
python mlp_wahab/evaluate.py \
    --model-dir ./mlp_wahab/trained_models/approach1 \
    --approach 1 \
    --data-root ./01_naive_all_layers
```

### Integration with SpecEE (requires GPU)

The trained `.pth` files are drop-in replacements for SpecEE's predictors. Both approaches use the same 12-feature input and identical MLP architecture as the original SpecEE models, so **no code changes** are needed in SpecEE.

#### Where the trained models live

```
EarlyExit/mlp_wahab/trained_models/
  approach1/model{0-31}.pth   # Naive baseline (contiguous gaps)
  approach3/model{0-31}.pth   # Augmented gaps (simulates dynamic scheduling)
```

#### Step-by-step (for whoever has a GPU)

1. **Copy the models** into SpecEE's expected directory structure. Each approach gets its own folder alongside the original `llama-7b/` predictors:

   ```bash
   # From the project root (where EarlyExit/ and SpecEE/ are siblings)
   cp -r EarlyExit/mlp_wahab/trained_models/approach1 SpecEE/SpecEE_cloud/llama-7b-wahab-a1
   cp -r EarlyExit/mlp_wahab/trained_models/approach3 SpecEE/SpecEE_cloud/llama-7b-wahab-a3
   ```

2. **Run accuracy benchmarks** — compare our models against the originals:

   ```bash
   cd SpecEE/SpecEE_cloud

   # Original SpecEE predictors (baseline to compare against)
   CUDA_VISIBLE_DEVICES=0 python EEInference.py \
       --base-model-path meta-llama/Llama-2-7b-chat-hf \
       --draft-model-path yuhuili/EAGLE-llama2-chat-7B \
       --predictor-path ./llama-7b \
       --dataset commonsenseqa --task accuracy

   # Our approach 1 (naive baseline)
   CUDA_VISIBLE_DEVICES=0 python EEInference.py \
       --base-model-path meta-llama/Llama-2-7b-chat-hf \
       --draft-model-path yuhuili/EAGLE-llama2-chat-7B \
       --predictor-path ./llama-7b-wahab-a1 \
       --dataset commonsenseqa --task accuracy

   # Our approach 3 (augmented gaps)
   CUDA_VISIBLE_DEVICES=0 python EEInference.py \
       --base-model-path meta-llama/Llama-2-7b-chat-hf \
       --draft-model-path yuhuili/EAGLE-llama2-chat-7B \
       --predictor-path ./llama-7b-wahab-a3 \
       --dataset commonsenseqa --task accuracy
   ```

3. **Run speed benchmarks** (optional, same pattern):

   ```bash
   CUDA_VISIBLE_DEVICES=0 python EEInference.py \
       --base-model-path meta-llama/Llama-2-7b-chat-hf \
       --draft-model-path yuhuili/EAGLE-llama2-chat-7B \
       --predictor-path ./llama-7b-wahab-a1 \
       --dataset mt_bench --task speed
   ```

4. **Datasets available for testing:**
   - Accuracy: `commonsenseqa`, `sst2`
   - Speed: `alpaca`, `gsm8k`, `mt_bench`, `sum`, `qa`, `humaneval`

5. **Tuning the exit threshold** (default 0.5): a lower threshold exits more aggressively (faster but riskier), higher is more conservative:

   ```bash
   CUDA_VISIBLE_DEVICES=0 python EEInference.py \
       --predictor-path ./llama-7b-wahab-a3 \
       --pred-thresholds 0.3 \
       --dataset commonsenseqa --task accuracy
   ```

#### What to look for in results

- **Accuracy** should be close to standard HuggingFace inference (no early exit). The script prints both for comparison.
- **Speed** is measured as tokens/second. The script prints SpecEE vs HuggingFace and the speedup ratio.
- Compare approach 1 vs approach 3 vs original `llama-7b/` predictors to see which handles the dynamic scheduling gap best.
